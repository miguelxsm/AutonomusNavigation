Important note about this computer
----------------------------------

Your default python3 comes from miniconda, and that breaks colcon/ROS builds.
Always start by sourcing the helper below from the repo root.

cd /home/miguel/Escritorio/ROB/project
source source_local_env.sh

The helper also forces FastDDS to use UDP only in local development to avoid
the noisy `RTPS_TRANSPORT_SHM Error` messages from shared-memory transport.

If you want me to inspect what a ROS command printed, use the logging wrapper:

cd /home/miguel/Escritorio/ROB/project
./run_ros_logged.sh ros2 launch rob_project relative_goal_test.launch.py goal_x:=0.3 goal_y:=0.0 goal_yaw:=0.0

This stores logs in:
- `runtime_logs/latest/console.log`
- `runtime_logs/latest/ros/`

Then you can tell me to inspect `runtime_logs/latest/console.log`.

Daily workflow for local development
------------------------------------

Before starting a fresh test, clean old ROS/Gazebo processes from previous
attempts:

cd /home/miguel/Escritorio/ROB/project
./cleanup_ros_local.sh

Or use the wrapper that already cleans and starts the full local stack:

cd /home/miguel/Escritorio/ROB/project
./start_local_validation.sh

Terminal 1 - build the project package:

cd /home/miguel/Escritorio/ROB/project
source source_local_env.sh
colcon build --packages-select rob_project
source install/setup.bash

If your terminal is zsh, use this instead of the previous line:

source install/setup.zsh


Terminal 2 - launch the usual TurtleBot3 simulator with GUI:

cd /home/miguel/Escritorio/ROB/project
source source_local_env.sh
ros2 launch turtlebot3_gazebo turtlebot3_world.launch.py

Other useful simulator worlds:

ros2 launch turtlebot3_gazebo empty_world.launch.py
ros2 launch turtlebot3_gazebo turtlebot3_house.launch.py


Terminal 3 - teleop check:

cd /home/miguel/Escritorio/ROB/project
source source_local_env.sh
ros2 run turtlebot3_teleop teleop_keyboard

If W/A/S/D moves the robot in Gazebo, the local simulator is working.


Fastest full-stack validation
-----------------------------

This is the easiest way to check the whole local stack:

Terminal 1:

cd /home/miguel/Escritorio/ROB/project
source source_local_env.sh
colcon build --packages-select rob_project
source install/setup.bash
ros2 launch rob_project local_nav2_validation.launch.py headless:=true

Logged version:

./run_ros_logged.sh ros2 launch rob_project local_nav2_validation.launch.py headless:=true

This starts:
- TurtleBot3 simulation
- SLAM Toolbox
- Nav2

When that is up, use a second terminal to move the robot a little and build a
small map:

Terminal 2:

cd /home/miguel/Escritorio/ROB/project
source source_local_env.sh
ros2 run turtlebot3_teleop teleop_keyboard

Then test a simple relative goal in a third terminal:

Terminal 3:

cd /home/miguel/Escritorio/ROB/project
source source_local_env.sh
source install/setup.bash
ros2 launch rob_project relative_goal_test.launch.py goal_x:=0.3 goal_y:=0.0 goal_yaw:=0.0

Logged version:

./run_ros_logged.sh ros2 launch rob_project relative_goal_test.launch.py goal_x:=0.3 goal_y:=0.0 goal_yaw:=0.0

Important: if the goal node prints warnings like
`Ignoring unexpected goal response` or if it never advances, you probably have
old Nav2 / Gazebo processes still running. Execute `./cleanup_ros_local.sh`
and restart the three terminals from zero.

If your terminal is zsh, use:

source install/setup.zsh

If you want GUI instead of headless:

ros2 launch rob_project local_nav2_validation.launch.py headless:=false use_rviz:=true

Simpler wrapper:

./start_local_validation_gui.sh

Teleop while Nav2 stack is running
----------------------------------

When `local_nav2_validation.launch.py` is active, do not publish teleop directly
to `/cmd_vel`, because Nav2 also owns the command path. Use this wrapper so the
keyboard commands go through `/cmd_vel_nav` and the velocity smoother:

cd /home/miguel/Escritorio/ROB/project
./teleop_local_nav.sh
